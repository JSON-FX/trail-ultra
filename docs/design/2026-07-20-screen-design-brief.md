# trail-ultra — Screen Design Brief (hand-off to a design agent)

**Purpose.** You are a **design agent**. Produce screen mockups for two surfaces of trail-ultra — the **iOS runner app** and the **Admin web console** — all in one consistent visual system. This document is self-contained: it tells you the design language, the product, and every screen with its content, actions, and states.

---

## 1. Design language — getdesign `apple` (summary; `apps/mobile/DESIGN.md` is authoritative)

Apple-marketing aesthetic: calm, spacious, high-contrast, hairline-delineated, **Action Blue** accent, **pill** primary buttons, minimal chrome/shadows.

| Token | Value | Use |
| --- | --- | --- |
| primary (Action Blue) | `#0066cc` | primary buttons, links, selected states, prices/accents |
| primaryFocus | `#0071e3` | pressed/focus |
| onPrimary | `#ffffff` | text on primary |
| ink | `#1d1d1f` | primary text |
| inkMuted | `#7a7a7a` | secondary text |
| inkFaint | `#cccccc` | placeholder |
| canvas | `#ffffff` | page background |
| parchment | `#f5f5f7` | cards / grouped background |
| hairline | `#e0e0e0` | borders, inputs, dividers |
| danger | `#ff3b30` | destructive / errors (iOS system red) |

- **Type:** SF Pro Display (headings, tight tracking) / SF Pro Text (body). Large bold titles; generous line-height.
- **Radii:** pill `9999`, lg `18`, md `11`, sm `8`.
- **Spacing scale:** 4 · 8 · 12 · 17 · 24 · 32.
- **Buttons:** primary = Action-Blue **pill**, white text; secondary = text/link in Action Blue; destructive = red text.
- **Inputs:** 1px hairline, radius `md`, white fill, generous padding.
- **Surfaces:** white canvas, `parchment` grouped cards, hairline dividers, essentially no drop shadows.
- **iOS chrome:** bottom tab bar (Action-Blue active tint), large screen titles, back as "‹ Label".
- **Web chrome:** left sidebar nav (icons + labels, Action-Blue active), slim top bar (org name + user/role menu), roomy content column, data tables with hairline rows. Same tokens as iOS.

## 2. Product context (one read)

trail-ultra is a **multi-organization** trail-run & ultra platform for Mindanao, Philippines. **Runners** browse a **marketplace** of events across organizations, open an event, register (core details + the org's custom fields + waiver + add-ons), pay in **₱ (PHP)** via PayMongo (Card / GCash / Maya), and carry a **signed QR ticket that works offline**. **Race directors** (organizations) run their events from the **Admin web** — create events, manage registrations and payments, check runners in at the start line, and see settlement. Data is siloed per organization; a runner has one global account.

- **Money:** always ₱ with 2 decimals, e.g. `₱1,500.00` (stored as integer centavos).
- **Roles:** `user` (runner) · `marshal` (check-in) · `editor` (event content) · `admin` (org operations) · `super_admin` (platform).
- **Event statuses:** `draft` (hidden) · `open` · `almost_full` · `closed` · `completed` · `cancelled`; **rescheduled** is shown when an event has a prior date recorded (still open, new date).
- **Registration/payment statuses:** `pending` · `paid` · `refunded` · `cancelled`.

## 3. iOS runner app — screens

Portrait, bottom tab bar: **Events · Orgs · My Races · Profile**. Every data screen needs four states — **loading** (skeleton), **empty**, **error** (retry), **offline** (banner; cached where available). Design each state where noted.

1. **Splash** — logo on canvas while the session restores. Minimal.
2. **Sign in** — email + password (hairline inputs), Action-Blue **pill** "Sign in", "Apple · Google · Facebook — coming soon" caption, "Create an account" link.
3. **Sign up / verify email** — name/email/password; then an "verify your email" confirmation state.
4. **Events — Marketplace** *(home tab)* — vertical list of **event cards across all organizations**. Card = hero image, event name, place · date, a **status badge** (Open / Almost full / Cancelled / Rescheduled), and the **organization name at the bottom of the card**. A search field (by name/place) at top. Show loading (skeleton cards), empty, and offline states.
5. **Event page** — swipeable **image gallery** (paged) at top; event name; a tappable **org header** (org photo + name → org page); **description**; meta row (place/region · date · elevation gain · cutoff); a **"Pick a distance"** list of categories, each with label, distance, **₱ price**, and slots-left; a persistent bottom **"Register"** primary pill. **Status banner** variants: a red "This event was cancelled" banner with the Register button disabled; a blue "Rescheduled — new date … (was …)" banner with Register enabled.
6. **Orgs** *(tab)* — vertical list of organizations; row = org photo (rounded), org name, and a subtle secondary line (region / "N events"). Loading + empty states.
7. **Org page** — **Facebook-profile style**: a wide **banner** image, an overlapping circular **org photo**, org name, an **about** paragraph, then a section header "Events" and a list of that org's **event cards** (same card as §4 but without the redundant org name). Design the header prominently.
8. **Register** — screen titled "Register — <category>". Renders the org's **dynamic fields** by type (text, number, **select** as chips, checkbox as switch, date as `YYYY-MM-DD` text); an **Add-ons** section (toggle rows with ₱ prices and a check); a **waiver** switch ("I accept the event waiver"); a **Total** row (₱, updates with add-ons); and a **"Register"** pill. Show inline validation errors in red.
9. **Payment** — summary (event — category, **₱ total**) and a **"Pay now"** pill that opens an in-app checkout sheet. Then a **Pending** state ("Waiting for payment confirmation…", spinner, "Check again"/"Retry"), and a **Confirmed** state ("Payment confirmed" + "View ticket" pill). (Payment provider page itself is a sandbox/PayMongo page — not your design.)
10. **Ticket** — the race pass: event name, category, a large **QR code**, a monospaced **Ref**, and "Show this QR at check-in. Works offline." If the event was cancelled/rescheduled, show the same lifecycle banner at top. Must read well offline (no reliance on remote images).
11. **My Races** — list of **all** the runner's registrations across every org; each row = event name, category, and a status pill (**Paid** / **Pending**). Tapping Paid → Ticket; Pending → resume Payment. Empty state ("No registrations yet").
12. **Profile** — editable fields (full name, bib name, city — hairline inputs), a **Save** pill, and a **Sign out** (red text). (No "switch organization" — that concept is gone.)

## 4. Admin web console — screens (MVP)

Desktop, **left sidebar** + top bar. Sidebar sections adapt to role. Same apple tokens; data-dense but airy, hairline tables, Action-Blue primary actions, red for destructive. Org admins see **only their org**; super_admin gets the platform section and an org switcher.

**Sidebar (org admin):** Dashboard · Events · Registrations · Payments · Check-in · Settings.
**Sidebar (super_admin, adds):** Organizations · Commission · Payouts.

1. **Login** — centered card on canvas: email + password, Action-Blue pill "Sign in". (Admin/staff are provisioned; no self-signup.)
2. **Dashboard** — KPI tiles (fill rate %, gross revenue ₱, paid registrations, upcoming events) + charts: sign-ups over time (line), category breakdown (bar), revenue. Date-range control. Design the empty/first-run state too.
3. **Events list** — table of the org's events: name, date, status badge, categories count, fill (taken/total), gross ₱. Primary "**Create event**" button. Per-row actions menu including **Edit**, **Reschedule**, **Cancel**.
4. **Create / Edit event** — a form: name, place, region, **date + flag-off time**, elevation gain, cutoff hours, **hero image + gallery** (upload), **description** (rich-ish text), status. Sub-editors on the same page (or tabs): **Categories** (code 100k/50k/21k/10k, label, distance, ₱ base price, slots_total) and **Add-ons** (name, ₱ price). Save / Cancel.
5. **Reschedule / Cancel modals** — *Reschedule*: new date + optional note ("Moved due to weather") → sets the new date and records the prior one (the app shows "Rescheduled — was …"). *Cancel*: confirmation + optional note → sets status `cancelled` (the app shows a Cancelled banner; registrations are untouched; refunds are handled from Payments). Red confirm for cancel.
6. **Custom registration fields** — an editor for the event's typed fields: a reorderable list of fields (key, label, **type** ∈ text/number/select/checkbox/date/file, **required** toggle, **options** for select), with add/edit/delete. This is what renders as the runner's dynamic Register form.
7. **Registrations table** — filterable (event, category, status pill, search by name), columns: runner name, category, status (pending/paid/refunded), ₱ paid, registered-at; expandable to show **custom field answers**. Export action.
8. **Registration detail** — one registration: runner + profile info, **custom data**, category/add-ons, payment (method, ₱, status), and a **Refund** action (destructive, admin-initiated).
9. **Payments** — list/table of payments with status, ₱ amount, method, platform fee, net-to-org; filters; **Refund** action. (Read-through of what settlement will aggregate.)
10. **Race-day check-in** — a **QR scanner** view (large camera frame) with a live result panel: **Valid → checked in** (green), **Already checked in** (amber), **Invalid ticket** (red). A manual **lookup** (bib/name) fallback. Designed for one-handed, high-contrast, fast use at a 4 AM start line; must convey it works offline and syncs later. (Marshal role sees only this.)
11. **Settings** — org profile the admin can edit: name, **logo**, **banner**, **description/about**, brand color, contact. (These feed the runner app's Org page.)

**super_admin (platform) screens:**
12. **Organizations** — table of all orgs (name, slug, status, commission %, events, gross); "**Provision organization**" flow (create org + its first admin).
13. **Commission** — set each org's commission rate (%). Simple table with inline edit.
14. **Payout statements** — per-org settlement: period, gross, fees, refunds, **net owed**, status (open/paid), and a **Mark as paid** action.

## 5. Cross-cutting design notes

- **States on every data screen:** loading (skeleton, not spinners where a list is expected), empty (friendly one-liner + optional CTA), error (message + retry), offline (a slim banner; iOS shows cached tickets/My Races).
- **Status color language:** Open = neutral/ink; Almost full = amber; Cancelled = `danger` red; Rescheduled = Action-Blue info; Paid = Action-Blue (or a blue tint pill); Pending = parchment/muted.
- **Money:** ₱ with two decimals everywhere; right-align in tables.
- **Accessibility:** WCAG AA contrast, Dynamic Type / scalable text, VoiceOver/ARIA labels, ≥44pt tap targets (iOS), visible focus (web).
- **Reusable components to define once (style tile):** primary pill button, secondary/text button, destructive button; hairline text input, select-chip, switch; card (event card variants: marketplace vs org-page), status badge (pill + banner), avatar/banner, data table row, modal, KPI tile, sidebar item, tab bar item.
- **Consistency:** the two surfaces must look like one product — same palette, type, radii, spacing, button language. The web is the "desktop expression" of the same apple system, not a different theme.
